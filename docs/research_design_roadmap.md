## 1. 研究全体像

### フェーズ構成

```
Phase 1（卒論・完了）         Phase 2（修論・進行中）
─────────────────────        ──────────────────────────────────────
  IRL による予測モデル   →       RL による介入シミュレーション
```

### 2.2 MDP 定義（Phase 2 拡張版）

#### 状態空間 $S$

| カテゴリ | 特徴量 | 備考 |
|--------|--------|------|
| **ミクロ（個人）** | 直近活動間隔、レビュー負荷、承諾率、応答速度 | 卒論10次元ベース |
| **ミクロ（個人）** | 過去コメント数、知識スコア（関与サブシステム数） | 拡張候補 |
| **マクロ（プロジェクト）** | タスク滞留数、リリースサイクル位相、全体承諾率 | 新規追加 |
| **知識転移** | 他プロジェクトでの経験ベクトル | Phase 2 後期 |

#### 行動空間 $A$

```
A = {Recommend(developer_id, task_id), NoAction}
```

- 特定の開発者に特定のタスク（レビュー/コミット）を推薦
- NoAction: 今ステップは推薦しない

#### 報酬関数 $R$

```
R(s, a) = Σ_i [IRL_reward(developer_i, s)] - λ · workload_penalty(a)
```

- **主報酬**: 全開発者の継続確率（IRL出力）の総和
- **負荷ペナルティ**: 特定開発者への集中を自動抑制（IRL内の負荷重みで実現）
- **λ**: ペナルティ強度の調整パラメータ

#### ステップ定義（イベント駆動型）

```
t = 0: 初期状態の設定
t = k: 新規タスク発生イベント
  → エージェントが推薦対象開発者を選択
  → 開発者がIRLモデルに従い承諾/拒否を確率的に決定
  → 状態を更新（直近1ヶ月窓でスライド集計）
t = k+1: 次のイベント発生まで待機
```

### 2.3 PU 学習の統合

```
観測データ                       PU 学習
──────────────                  ─────────────────────────────
貢献あり（ラベル=1）  ──────────▶  潜在的な貢献可能性を推定
貢献なし（ラベル=0）              （未観測ポジティブを識別）
  └ 実際に拒否した？              ↓
  └ タスクが来なかった？          修正済みラベルで IRL を再学習
```

---

### 2.4 破壊的忘却の防止戦略

| 手法 | 概要 | 適用場面 |
|------|------|---------|
| EWC (Elastic Weight Consolidation) | 重要パラメータに制約を課す | プロジェクト間の逐次学習 |
| Replay Buffer | 過去データのミニバッチを混入 | 軽量で実装しやすい |
| Progressive Networks | プロジェクトごとに列を追加 | 汎化性能の検証向け |

→ まず **Replay Buffer** で実装し、性能が不十分なら EWC へ移行。

## 4. 評価計画

### 4.1 再現性評価（RQ3）

| 指標 | 計測方法 | 目標値 |
|------|---------|-------|
| AUC-ROC | 史実の推薦 vs エージェント選択 | > 0.75 |
| F1スコア | 閾値最適化後 | > 0.70 |
| 推薦一致率 | Top-K推薦の史実との重複 | K=3で> 50% |

### 4.2 シミュレーション評価（RQ4）

| 指標 | ベースライン（史実） | AI介入後 |
|------|-------------|---------|
| LTC存続数（12ヶ月後） | 実測値 | 改善率を計測 |
| 変動型開発者の承諾率 | 実測値（33.3%が変動型） | 改善率を計測 |
| 最大個人負荷（過集中度） | 実測値 | 低下を確認 |
| 平均タスク応答時間 | 実測値 | 短縮を確認 |

## 5. リスクと対策

| リスク | 深刻度 | 対策 |
|-------|-------|------|
| RL が収束しない | 高 | 報酬のスケーリング調整、シンプルなアルゴリズムから開始 |
| シミュレータが史実と乖離 | 中 | 承諾確率モデルの精度検証を M1 で徹底する |
| データ取得が困難（他プロジェクト） | 中 | Gerrit API での取得を M4 前に確認 |
| PU学習の効果が出ない | 低 | 通常の教師あり学習と比較し、有効性を判断してから採用 |
| 計算コストの超過 | 低 | GPU サーバの利用計画を指導教員と確認 |

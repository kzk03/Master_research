# 修士研究 計画書

## 全体コンセプト

卒論で構築した「IRLによる報酬関数」を土台に、RLによる**タスク推薦・貢献予測システム**へと発展させる。
分析軸を「**人（開発者）× ファイルパス**」の2軸で捉えることで、従来研究との差別化を図る。

```
卒論（IRL）
  └─ 開発者の意思決定をMDPとしてモデル化
  └─ 報酬関数（継続確率）を推定 → 修論で再利用

修論（RL）
  └─ モード①：タスク発生時に「誰に振るか」を推薦（能動的介入）
  └─ モード②：「このタイミング × このタスクなら誰が貢献するか」を予測（事前スコアリング）
  └─ モード③：過去の行動をどれだけ模倣・再現できるか（時系列考慮の効果検証）
  └─ 評価軸：あるべき姿（RL推薦）vs 実際の歴史（再現）
```

---

## 卒論の成果と課題

### 成果
- 開発者の行動を「時系列の意思決定プロセス」としてモデル化し、IRLで評価
- 予測期間（短期・長期）によって重要特徴量が変化することを確認
- 全開発者の約33.3%を占める「変動型開発者」の特定

### 残された課題（修論での対応方針）

| 課題 | 修論での対応 |
|------|------------|
| 報酬関数がプロジェクト共通（全開発者同一） | 個人・プロジェクト横断モデルへ拡張 |
| 単一プロジェクト（OpenStack/Nova）依存 | Cross-Project検証、マクロ状態にプロジェクト依存関係を追加 |
| 状態・行動特徴量が少ない | ファイルパスの知識を含む特徴量拡張 |
| 行動がレビューのみ | コミット等まで拡張 |
| 予測にとどまる | RLによる介入・推薦システムへ |

## 2軸アプローチ：人 × ファイルパス

### 従来研究との位置づけ

```
従来研究（内容込み予測）
  └─ コードの内容・ファイルパスから「誰が来るか」を予測（受動的）

本研究の差別化
  └─ 「誰が来るか」ではなく「誰を推薦すべきか」をRLで最適化（能動的）
  └─ 報酬 = 全開発者の継続確率の総和（卒論IRL由来）
```

### ファイルパスを加えることで見えること

- **人ベース**: 開発者Aは今どのくらい活発か、負荷は高いか
- **パスベース**: このファイルに詳しい（過去に触れた）開発者は誰か
- **2軸の組み合わせ**: 「このファイルに詳しく、かつ今余裕のある開発者」を推薦 → 精度・妥当性が向上

## MDP設計

### 状態 S

**ミクロ（個人レベル）**
- 開発者の現在の負荷（未処理タスク数）
- 直近の活動間隔（どれくらい前に最後に貢献したか）
- 知識（担当ファイルパスの履歴、卒論の特徴量ベースに拡張）

**マクロ（プロジェクトレベル）**
- プロジェクトの繁忙期（リリースサイクル）
- 全体のタスク滞留数
- プロジェクト間依存関係（プロジェクトAでの経験がプロジェクトBに与える影響）

**タスクレベル（新規追加）**
- 対象ファイルパス
- タスクの種類（レビュー / コミット / 他）
- タスクの発生タイミング

### 行動 A

特定の開発者へのタスク推薦（Recommend）

- 入力: 発生したタスク（ファイルパス + 種類 + タイミング）
- 出力: 推薦する開発者（1名 or ランキング）

### 報酬 R

全開発者の「継続確率（IRLの報酬関数出力値）」の総和

```
R = Σ P_continue(developer_i)
```

**暴走防止（必要に応じて）**
- 特定の開発者にタスクを集中させると報酬にペナルティ → 自動的な負荷分散

## 3つのモード

### モード①：タスク発生時推薦（Task-triggered Recommendation）

> タスクが来たタイミングで「誰に振るべきか」をリアルタイムに推薦する

- イベント駆動型: タスク発生を1ステップとして処理
- RLエージェントが状態Sを参照し、最も継続確率を高める開発者を推薦
- 実用的なシステムとして設計可能

### モード②：貢献可能性予測（Contribution Prediction）

> 「このタイミングでこのタスクが来たら、この人は貢献してくれるか」を事前に予測する

**卒論からの拡張**

```
卒論:   P(contribute | developer, time_window)
        → N ヶ月後にこの人は貢献してくれるか？

修論:   P(contribute | developer, time_window, file_path, task_type)
        → N ヶ月後に「このパス」の「この種類のタスク」が来たら、この人は取り組んでくれるか？
```

卒論と同様に期間（例: 1ヶ月後）を設定した上で、**ファイルパスとタスク種別を条件として追加**することで、より具体的な貢献可能性を予測できる。

**予測の条件軸**

| 軸 | 卒論 | 修論 |
|----|------|------|
| 開発者 | ✅ | ✅ |
| 予測期間（N ヶ月後） | ✅ | ✅ |
| ファイルパス | ❌ | ✅（新規） |
| タスク種別 | 一部 | ✅（新規） |

**具体的なクエリ例**

- 「1ヶ月後に `nova/compute/manager.py` に関するレビューが来たら、開発者Aは取り組むか？」
- 「次のリリースサイクルでこのモジュールに誰が貢献しやすいか？」

**PU学習との組み合わせ**

- 観測されなかった（タスクが発生しなかった）ケースをネガティブとみなさず、潜在的なポジティブとして扱う
- 使い方の例:
  - タスク割り当て前のスクリーニング
  - 「このパスに詳しくて、来月余裕がある開発者」を事前にリストアップ

### モード③：行動模倣・再現精度の検証（Behavioral Cloning Evaluation）

> RLが時系列を捉えることで、過去の開発者行動をどれだけ模倣・再現できるかを検証する

- **何をやるか**: 学習済みRLエージェントに過去と同じ状態を与え、実際に起きた行動（誰が担当したか）をどれだけ再現できるかを測定
- **なぜ面白いか**: 時系列を無視した単純な機械学習（例: ある時点のスナップショットで予測）と比較し、**RLによる時系列の考慮が予測精度を向上させることを実証**できる
- **ベースラインとの比較**:

  | モデル | 時系列の扱い | 想定精度 |
  |--------|------------|---------|
  | スナップショット予測（従来） | 無視 | 低 |
  | IRL（卒論） | 考慮（MDP） | 中 |
  | RL（修論） | 考慮 + 最適化 | 高（仮説） |

- **位置づけ**: モード①②が「これからどうすべきか」の提案なのに対し、モード③は「過去をどれだけ説明できるか」の検証 → 研究の妥当性担保として機能する

## 評価設計

### 再現評価：あるべき姿 vs 史実

RLエージェントが推薦した開発者 ↔ 実際に担当した開発者 の一致率を測定

| 結果 | 解釈 |
|------|------|
| 一致率が高い | モデルが実際の意思決定を再現できている（妥当性の証拠） |
| 一致率が低い | RLが「より良い割り当て」を提案している（改善提案として解釈） |

一致しない部分を**なぜRLが異なる推薦をしたか**で説明できると、研究の貢献が明確になる。

### シミュレーション評価

- 介入後の変動型開発者の承諾率の変化
- 開発者の離脱率（継続確率の低下）の抑制効果
- 負荷分散の改善度（特定開発者への集中度の低下）

## モデルの工夫点

### PU学習（Positive-Unlabeled Learning）

貢献しなかった開発者を単純に「ネガティブ」とみなさず、**潜在的な貢献可能性（未観測のポジティブ）** として扱う。

ポジティブの定義候補:
- 過去に同種ファイルに貢献した履歴がある
- 活動間隔が短い（アクティブ状態）
- タスクの種類が得意分野に合致している

### 破壊的忘却の防止

プロジェクト横断学習において、過去のプロジェクトでの知識を維持しながら新しいプロジェクトに適応するための工夫（Continual Learning）。

### Human-Agent Collaboration（将来ビジョン）

人間とAIエージェントが混在するチームにおけるリソース管理の最適化。RLエージェントが推薦し、人間が承認・修正するインタラクティブなループ設計。

---

## シミュレーションの「ステップ」設計

2つの候補:

1. **イベント駆動型**: タスクが発生した瞬間を1ステップ（モード①向け）
2. **期間集約型**: 卒論と同様に1ヶ月単位で状態を更新（モード②・評価向け）

両方を組み合わせ、タスク単位の推薦と月次の評価を分離して設計することも可能。

## メモ: コミット・PR作成の扱いについて（未決定）

レビュー以外の行動（コミット・PR作成・マージ）をどこまで取り込むかの検討メモ。

### 追加できる行動の種類

| 行動 | Gerrit での対応 | 取得コスト |
|-----|--------------|----------|
| PR作成（Change作成） | `owner_email` として既にデータにある | 低（既存データから抽出） |
| コミット（Revision） | 各 Change の `revisions` フィールド | 低〜中 |
| マージ | `status: MERGED` + `submitter` | 低 |

### 用途別のメリット・デメリット

**① 状態 S の特徴量として追加する場合**

メリット:
- 「最近コミットが多い開発者は忙しい」など開発者の状態をより多角的に捉えられる
- IRL の特徴量が豊かになり報酬関数の精度向上に直結
- 実装コストが低い（`common_features.py` に追加するだけ）
- 卒論との連続性を保ちやすい

デメリット:
- 特徴量が増えすぎると IRL の学習が不安定になる可能性
- Gerrit では「コミット」と「Change 作成」の分離に少し手間がかかる

**② RL のアクション空間に追加する場合（レビュー以外も推薦対象にする）**

メリット:
- 「レビュー・コミット・PR作成を誰に振るか」という広い推薦システムになる
- Human-Agent Collaboration の文脈で実用的なシステムに近づく
- 研究の主張が広がる

デメリット:
- 行動空間が `開発者数 × タスク種別数` に膨らむ（例: 50人 × 3種 = 150次元）→ RL の学習が難しくなる
- 「コミットを推薦する」という設定の現実性が問われる（コミットは自発的な行動）
- 卒論との連続性が薄れる

### 方針案（暫定）

```
優先: 状態 S の特徴量として追加
  → 単一プロジェクト RL 検証が終わった後に判断
  → 実装コスト低・精度向上効果が見込める

保留: アクション空間への追加
  → RL 検証完了後に設計を検討
  → 行動空間の爆発問題を先に解決する必要あり
```
